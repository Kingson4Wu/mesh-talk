## 分部分说明

### 一、节点完全对等，开关只是用户配置

PC-1 / PC-2 / PC-3 跑的是**同一份程序**，能力上没有任何差异。区别只在两个独立的用户可调开关：

| 开关 | 打开后做什么 | 关闭的影响 |
|---|---|---|
| 中继/HUB 开关 | 本机额外起一个 WS 信令服务，监听在自己的局域网 IP 上，能接待手机/其他设备 join | 关闭后这台机器不能当跳板，但**不影响它自己正常收发消息、参与同步** |
| 下载服务开关 | 本机额外起一个 HTTP 静态服务，暴露 PWA 资源 + 生成二维码 | 关闭后这台机器不能给手机提供下载入口，跟通信能力无关 |

PC-3 两个开关都没开，但它依然会被 PC-1/PC-2 通过**局域网 UDP 发现**机制认识，进而被记入 presence 名录；它和 PC-1、PC-2 之间的**常驻 TCP+Noise 连接照常存在**——这条连接本来就是局域网 mesh 同步的一部分，跟中继开关无关。

### 二、全程局域网，没有公网层

信令服务**就跑在某台开了中继开关的 PC 本机上**，监听的是它自己的局域网 IP（如 `ws://192.168.1.10:9001`），手机连的也是这个地址。整张图里**不存在任何走出局域网的流量**。

### 三、证书问题全梳理

| 环节 | 协议 | 需要证书？ | 说明 |
|---|---|---|---|
| 扫码下载 PWA | `http://`局域网IP | 不需要 | 没有CA签发证书的可能，也用不上https |
| PWA"安装"成App | Service Worker注册 | 实际上需要安全上下文 | http局域网IP不算安全上下文，可能导致装不上，只能当网页用——需实测确认 |
| 信令交换 | `ws://` | 不需要 | 页面是http origin，用ws不冲突，不会被mixed-content拦截 |
| WebRTC DataChannel底层加密 | DTLS | 浏览器自动生成临时自签名证书 | 全自动、完全透明，只管管道加密，不涉及身份判断 |
| 应用层身份认证 | Noise XX | 完全不需要证书 | 用app自己的Ed25519/X25519长期密钥做双向认证，这是整个系统唯一真正的信任根 |

唯一会用到正经CA证书的场景：把PWA部署到固定公网域名（如GitHub Pages）解决"换IP=换身份"问题，那时https origin会强制信令走`wss://`。**纯局域网部署完全不需要任何证书。**

### 四、消息双向流转的核心机制（本次新增重点）

**关键认识：消息能否送达，只取决于"发送方到接收方之间是否存在一条由常驻连接组成的接力路径"，跟节点是不是HUB、是否开了中继开关完全无关。**

每一段连接（PC↔PC、PC↔HUB、HUB↔手机）都是**持续打开、双向的常驻连接**。任意一端写入新事件后，会在已经打开的连接上推送一个"有更新"的轻量通知（而非等下一个心跳周期），对方收到立即触发同步、接收新事件。

这使得三种消息流向都自然成立，且完全对称：
- **非中继PC → 手机**：PC-3写消息 → 经PC-1(HUB)接力 → 手机被动收到
- **手机 → 非中继PC**：手机写消息 → 经PC-1(HUB)接力 → PC-3被动收到
- **手机 → 手机**：经一个或两个HUB接力（视两台手机连的是否同一HUB），原理跟上面完全一致

发送方不需要知道完整路径，只需要把消息写进本地日志、推给下一跳；接收方全程被动接收。路径上每一段都是早已建立好的常驻连接，不需要为某条具体消息临时新建连接。

### 五、用户层完全对等、中继过程透明

手机和任何一台 PC 在以下三层是完全一样的：
- **数据模型**：同一套事件日志格式，手机写的消息和PC写的没有任何区别
- **身份**：手机的身份就是它自己的密钥对，跟PC是同一种东西，没有二等公民设计
- **体验**：用户在手机上看到的好友、聊天记录、在线状态，跟PC上看到的是同一份最终收敛的数据，且消息可以双向、主动地在任意两个节点间流转

中继这层东西存在的唯一原因，是**手机这个浏览器环境天生没有监听端口、不能被局域网发现协议直接发现**，需要借一台已经在网络里的PC当接入跳板——这纯粹是"怎么把字节送到对方那"的技术细节，跟"连上网之后用户能看到什么、能做什么、能不能主动发消息给谁"完全是两件事，对用户而言整个中继和接力过程是完全透明的。

### 六、HUB故障切换（仍是当前协议缺失、需要补的扩展能力）

手机本地缓存的"备用HUB列表"来自第一次同步presence时筛出的"中继开关=ON"节点；HUB失联后直接用缓存地址重新走信令流程连新HUB，**不需要重新扫码**。这部分逻辑文档原版协议里没有实现（属于已知缺陷，§7第2点），但思路和数据来源都已经在现有架构里齐备，补起来不需要改动事件日志和加密层。

## 整体改动点（相对之前的架构图）

```
之前: 二维码/图标 → http://192.168.1.10:8080
现在: 二维码/图标 → http://meshtalk-pc1.local:8080

```

信令地址同理：`ws://meshtalk-pc1.local:9001` 而不是裸IP，这样手机缓存的"备用HUB列表"里存的也应该是 `.local` 域名而不是IP，IP变化时整个系统（下载入口、信令、备用HUB切换）都不受影响。

## 具体要做的几件事

### 1. PC端：注册一个稳定的 mDNS 主机名/服务名
每台开了"中继开关"或"下载服务开关"的PC，启动时要通过 mDNS（Bonjour/Avahi）广播一个**固定不变**的服务名，比如基于这台机器的用户起的设备名生成 `meshtalk-{设备名}.local`。这个名字应该是用户首次设置时定下来的，**重装系统、换网卡都不应该让它变**——因为它现在替代了IP，成了新的"稳定标识"。

实现上，各平台的 mDNS 库：
- Windows：需要应用自己跑 mDNS responder（Windows原生对mDNS支持不算好，通常得用 Bonjour for Windows 或自带的 mDNS 库，比如 `mdns` npm包配合 Tauri/Electron的本地服务）
- macOS：系统自带 Bonjour，体验最好
- Linux：依赖 Avahi（大多数发行版默认装了）

### 2. 手机端：浏览器能不能直接解析 `.local`
这是**最关键的现实问题**——浏览器本身**不直接支持** mDNS 解析。`http://xxx.local` 这种地址，能不能打开取决于**操作系统的网络栈**是否支持 mDNS，而不是浏览器：

| 平台 | 系统层mDNS支持 | 浏览器里访问`.local`地址 |
|---|---|---|
| macOS | 原生支持(Bonjour) | ✅ 能直接解析 |
| iOS | 原生支持 | ✅ 能直接解析 |
| Windows | 默认**不支持**(除非装了Bonjour) | ❌ 大概率失败 |
| Android | 部分支持,但碎片化严重(不同厂商ROM差异大) | ⚠️ 不稳定,经常失败 |

也就是说**你想解决的"手机端"恰恰是支持最差的一边**——尤其Android碎片化问题，是这个方案最大的现实风险，需要你提前用目标用户的实际机型测试。

### 3. 缓解 Android/部分场景解析失败的兜底方案
因为浏览器层mDNS支持不可靠，实用做法通常是**app自己做一层mDNS查询**，而不是依赖系统/浏览器：

```
手机PWA启动后:
  ① 先尝试缓存的上次成功地址(IP或.local域名)
  ② 如果失败,用 WebRTC的mDNS候选机制 或 浏览器能力受限场景下
     退化成"局域网IP段扫描"(类似PC端发现机制里的/24 fallback)
  ③ 找到能响应的HUB后,把这次解析出的实际IP缓存下来,
     下次优先用缓存的IP直连(快),mDNS只作为IP变了之后的"找回"手段
```

注意：纯网页/PWA环境下，**JS本身没有原生mDNS API**，所以这个"app自己查询"实际上只能借助 WebRTC 建立连接过程中,ICE candidate 协商阶段会用到的 `.local` mDNS candidate（Chrome等浏览器为了防止局域网IP泄露隐私，WebRTC的ICE candidate本身默认就会用一个随机`.local`地址代替真实IP）——但这个机制是给**WebRTC连接建立**用的，不能直接当成"任意访问某个http地址"的通用域名解析能力，二维码下载这一步（裸http访问）依然会受限于系统层mDNS支持。

### 4. 实际建议的落地顺序

考虑到Android/Windows的mDNS支持短板，建议这样分层：

1. **信令服务地址（ws://xxx.local:9001）**：优先用`.local`，这一层主要是PC之间、以及"手机已经连上一次之后"用，PC端（macOS/Linux支持好）之间用mDNS没问题；
2. **手机首次下载入口**：**不要完全依赖`.local`**，二维码里同时编码 `.local`域名 + 当前实际IP 两种地址，PWA加载逻辑里"先试.local，超时后自动回退到IP"，兼顾鲁棒性和体验；
3. **手机缓存的备用HUB列表**：每条记录同时存 `hostname.local` 和"最近一次成功连接时用的实际IP"，重连时**先试IP（更快，大概率还有效），失败再试.local（应对IP变化的情况），还失败再退化成局域网扫描**。

这样即使某些Android设备的mDNS解析完全不工作，至少"IP没变的情况下直连"和"扫描兜底"两条路还在，`.local`是作为**优化层**叠加上去的，不是单点依赖。

## 一句话总结

mDNS方案能解决"PC端IP变化导致备用HUB列表失效"这个核心问题（PC端系统层支持都不错），但**不能完全指望手机浏览器/系统层支持`.local`解析**，尤其Android要做好大概率失败的预案，所以实现上建议是"`.local`优先 + IP缓存 + 局域网扫描兜底"三层兼顾，而不是单纯把所有IP替换成`.local`域名就完事。
